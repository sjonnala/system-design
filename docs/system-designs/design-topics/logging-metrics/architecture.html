üóÑÔ∏è LOG STORAGE
Elasticsearch
Architecture:
  • Indices: Time-based (logs-2025.11.15)
  • Shards: 30 primary + 30 replica
  • Index Lifecycle: Hot ‚Üí Warm ‚Üí Cold ‚Üí Delete
  • Retention: 90 days searchable
# Index template { "index_patterns": ["logs-*"], "settings": { "number_of_shards": 3, "number_of_replicas": 1, "refresh_interval": "5s" }, "mappings": { "properties": { "timestamp": {"type": "date"}, "level": {"type": "keyword"}, "service": {"type": "keyword"}, "message": {"type": "text"}, "trace_id": {"type": "keyword"} } } }
Optimization:
  • Hot nodes: SSD storage (last 7 days)
  • Warm nodes: HDD storage (8-30 days)
  • Cold nodes: Archive (31-90 days)
  • Snapshot to S3: Long-term retention
Alternative: Loki (Grafana)
  • Cost-effective (indexes metadata only)
  • Stores logs in object storage
  • LogQL query language
  • Lower resource usage

üóÇÔ∏è DATA LIFECYCLE & RETENTION
Tiered Storage Strategy
Logs:
  • Hot (0-7 days): ES on SSD, full search
  • Warm (8-30 days): ES on HDD, reduced replicas
  • Cold (31-90 days): ES frozen, S3 snapshot
  • Archive (90+ days): S3 Glacier, compliance
Metrics:
  • Raw (15 days): Prometheus local
  • 5min avg (90 days): Thanos/S3
  • 1hr avg (1 year): Downsampled
  • 1day avg (5 years): Long-term trends
# Retention policy Logs total storage: 50TB ‚îú‚îÄ Hot: 5TB (Elasticsearch SSD) ‚îú‚îÄ Warm: 15TB (Elasticsearch HDD) ‚îú‚îÄ Cold: 30TB (S3 Standard) ‚îî‚îÄ Archive: 500TB+ (S3 Glacier) Metrics storage: 10TB ‚îú‚îÄ Raw: 2TB (Prometheus) ‚îî‚îÄ Long-term: 8TB (Thanos/S3)
Compliance & Auditing
  • Immutability: WORM storage for audit logs
  • Encryption: At rest (AES-256), in transit (TLS)
  • Access logs: Who queried what, when
  • Retention policies: GDPR, SOX, HIPAA compliance

üîê SYSTEM DESIGN CONSIDERATIONS

Scalability Strategies

  • Horizontal Scaling: Kafka partitions, ES shards
  • Data Partitioning: Time-based, service-based
  • Read Replicas: Scale query performance
  • Caching: Redis for hot data
  • Async Processing: Decouple ingestion from storage

Data Quality & Reliability

  • Schema Validation: Enforce log structure
  • Data Deduplication: Handle duplicate events
  • Backpressure Handling: Queue buffering
  • Dead Letter Queue: Handle failures
  • Monitoring: Data pipeline health

Security Measures

  • Authentication: API keys, mTLS, OAuth
  • Authorization: RBAC for queries
  • Encryption: TLS, at-rest encryption
  • Data Masking: PII redaction
  • Audit Trails: Access logging

Cost Optimization

  • Sampling: 10% sample for debug logs
  • Compression: gzip, snappy (5-10x reduction)
  • Tiered Storage: Hot/warm/cold/archive
  • Retention Policies: Auto-delete old data
  • Query Optimization: Limit time ranges

Observability Maturity Model: This design represents Level 4 - Full Observability with predictive analytics and AIOps capabilities
