üóÑÔ∏è DATABASE LAYER
Message Database (Cassandra)
Schema Design:
-- Messages Table (time-series) CREATE TABLE messages ( chat_id UUID, message_id TIMEUUID, sender_id UUID, content TEXT, encrypted_content BLOB, type TEXT, -- text, image, video, audio status TEXT, -- sent, delivered, read created_at TIMESTAMP, PRIMARY KEY (chat_id, message_id) ) WITH CLUSTERING ORDER BY (message_id DESC); -- Inbox Table (per user) CREATE TABLE user_chats ( user_id UUID, chat_id UUID, last_message_time TIMESTAMP, unread_count INT, PRIMARY KEY (user_id, last_message_time, chat_id) ) WITH CLUSTERING ORDER BY (last_message_time DESC);
Why Cassandra?
  • High write throughput (millions msgs/sec)
  • Time-series data optimization
  • Linear scalability
  • Multi-datacenter replication
  • No single point of failure
User Database (PostgreSQL)
-- Users Table CREATE TABLE users ( id UUID PRIMARY KEY, phone_number VARCHAR(20) UNIQUE, username VARCHAR(50) UNIQUE, email VARCHAR(255), public_key TEXT, -- for E2E encryption created_at TIMESTAMP ); -- Contacts Table CREATE TABLE contacts ( user_id UUID, contact_id UUID, added_at TIMESTAMP, PRIMARY KEY (user_id, contact_id) );
Features:
  • ACID transactions
  • Sharded by user_id
  • Read replicas for scaling

üîê ADDITIONAL SYSTEM CONSIDERATIONS

Security & Encryption

  • End-to-End Encryption: Signal Protocol, Double Ratchet
  • Transport Security: TLS 1.3 for all connections
  • Authentication: Phone number OTP, biometric
  • Key Management: Public key infrastructure
  • Perfect Forward Secrecy: Ephemeral key exchange
  • Message Integrity: HMAC signatures

Scalability Strategies

  • Horizontal Scaling: Stateless WebSocket servers
  • Database Sharding: By user_id and chat_id
  • Connection Pooling: 100K connections per server
  • Message Batching: Group multiple messages
  • Async Processing: Event-driven for non-critical tasks
  • Multi-Region: Geographic distribution

Data Consistency

  • Message Ordering: TIMEUUID for chronological order
  • Eventual Consistency: Cassandra with tunable consistency
  • Idempotency: Duplicate message detection
  • Conflict Resolution: Last-write-wins (LWW)
  • Delivery Guarantees: At-least-once delivery

Failure Scenarios & Recovery

  • WebSocket Failover: Auto-reconnect with exponential backoff
  • Database Failover: Multi-DC replication (30s RTO)
  • Message Queue: Persistent storage, replay capability
  • Network Partition: Offline message queue on client
  • Data Loss: Daily backups, point-in-time recovery
