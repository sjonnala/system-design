üì§ File Upload Flow (Chunked & Deduplicated)

  1. User Selects File: User drags file to desktop client or web UI
  2. Client Chunks File: Split file into 4MB chunks using content-defined chunking (Rabin fingerprint)
  3. Calculate Checksums: SHA-256 hash for each chunk
  4. Check Existing Chunks: Query metadata service: which chunks already exist?
    • ‚úÖ Chunk exists ‚Üí Skip upload (deduplication)
    • ‚ùå Chunk new ‚Üí Upload to block storage
  5. Parallel Upload: Upload new chunks in parallel (10 concurrent connections)
  6. Resumable: If connection fails, resume from last successful chunk
  7. Create Metadata: Save file metadata + chunk mapping to database
  8. Publish Event: Kafka event triggers: thumbnail generation, search indexing, sync notification
  9. Notify Clients: Push notification to all synced devices via WebSocket
üéØ Key Optimization: Deduplication can reduce storage by 50%. If user uploads file that already exists, only metadata is created!

Example: 100MB File Upload

File: report.pdf (100MB)
Chunks: 25 chunks √ó 4MB = 100MB

Chunking: ~2 seconds
Hash calculation: ~3 seconds
Dedup check: ~500ms (Redis)
Upload new chunks (assume 10 new): 10 √ó 4MB / 10 Mbps = ~3 seconds
Metadata save: ~100ms
Total: ~9 seconds for 100MB file

With 100% dedup (file exists): ~3 seconds (no upload!)

üì• File Download Flow

  1. User Requests File: Click download in web UI or sync request from desktop client
  2. Authenticate & Authorize: Verify user has permission to access file
  3. Fetch Metadata: Query metadata service for file chunks list
  4. Generate Signed URLs: Create time-limited S3/CDN URLs for each chunk (valid 1 hour)
  5. Parallel Download: Client downloads chunks in parallel from CDN (if cached) or S3
  6. Streaming: Client can start displaying/playing file while downloading (progressive download)
  7. Reassemble: Client concatenates chunks in correct order
  8. Verify Integrity: Validate SHA-256 checksums match
  9. Cache Locally: Store in client local cache for offline access
üöÄ CDN Advantage: 99% of downloads served from CDN edge (10-50ms latency), only 1% hit origin S3 (200-500ms)

üîÑ Real-time Synchronization Flow

  1. File Modified Locally: Desktop client detects file system change (via inotify/FSEvents)
  2. Calculate Delta: Compare with last synced version
    • Only changed chunks need re-upload
    • Delta sync saves bandwidth
  3. Upload Changes: Upload only modified chunks to block storage
  4. Update Metadata: Increment file version, update modified timestamp
  5. Publish Sync Event: Kafka event: file.modified
  6. Sync Coordinator: Determines which clients need notification
  7. WebSocket Push: Push notification to all active clients
    WS Message: {"type": "file.modified", "fileId": "abc", "version": 2}
  8. Clients Pull Changes: Active clients fetch delta: GET /sync/delta?since=version1
  9. Download New Chunks: Only download chunks that changed
  10. Apply Changes: Update local file system
  11. Conflict Resolution: If multiple clients modified simultaneously:
    • Last-Writer-Wins: Use timestamp
    • Create Conflict Copy: Save both versions (file.txt vs file_conflict_2025-11-14.txt)
    • Manual Resolution: User chooses which version to keep

üèóÔ∏è Complete Blob Storage System Architecture

Client Layer

  • Desktop: Windows, Mac, Linux clients
  • Mobile: iOS, Android apps
  • Web: React/Vue.js Progressive Web App
  • API: RESTful API + SDKs

Edge & Gateway

  • CDN: CloudFront, Fastly (200+ POPs)
  • Load Balancer: AWS ALB, NGINX
  • API Gateway: Kong, authentication/rate limiting

Core Services

  • Upload/Download: Handle file transfers
  • Metadata: PostgreSQL sharded by user_id
  • Sync: Real-time coordination
  • Sharing: Permission management
  • Search: Elasticsearch full-text
  • Preview: Thumbnail/transcoding

Storage Layer

  • Block Storage: S3, GCS, Azure Blob
  • Chunking: 4MB fixed-size or content-defined
  • Deduplication: 50% storage savings
  • Tiering: Hot/Warm/Cold (S3 IA, Glacier)

Caching & Queue

  • Redis: Metadata cache, sessions
  • Kafka: Event streaming, async processing
  • Topics: file.uploaded, file.modified, sync.trigger

Observability

  • Metrics: Prometheus + Grafana
  • Logs: ELK Stack (Elasticsearch, Logstash, Kibana)
  • Tracing: Jaeger distributed tracing
  • Alerts: PagerDuty, OpsGenie
